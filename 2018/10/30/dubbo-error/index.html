  • 定位原因

    当一个dubbo服务，先有consumer订阅，后有provider注册时
    且存在override协议，设置该dubbo协议的disable=true
    此时consumer端无法创建任何一个invoker，报出此异常

Dubbo异常信息：Not found exported service: com.xxx.FooService:20070.

场景：
    在一个应用中同时export多dubbo服务，且满足一下条件:
        1. 暴露服务的serviceName相同，但是服务实现不同
        2. 分别在不同的注册中心暴露
    例如:
        /dubbo/r1/com.xxx.FooService
        /dubbo/r2/com.xxx.FooService

    此时，unexport其中一个服务节点，再调用其他服务，就会报错。

  • 问题分析

    通过源码分析发现其根本原因在于Dubbo Protocol中的serviceKey
    首先我们看下DubboExporter

    当unexport时，exporterMap会移除export，而这个map的key就是serviceKey
    再来看下serviceKey的构成
    serviceKey = ${group} [/] ${interfaceName} [:] ${version}
    由此我们发现所有暴露的服务的serviceKey都是相同的
    所以当unexport其中某个服务，会根据serviceKey移除所有的export
    注: 所有服务共享一个端口暴露

  • 解决办法

    1. 设置group
    2. 设置version
      目的: 保证serviceKey不唯一

Dubbo异常信息：client reconnect to 10.19.133.29:20880 find error.

场景：
    当一个dubbo服务支持多机房的情况
        1. provider端本机房暴露服务
        2. consumer端多机房订阅服务
        3. 每个机房都有独立的注册中心
    假设
        1) 现在有一个com.xxx.FooService服务
        2) 这个服务在A机房暴露服务
        3) 同时有个消费者分别在A机房和B机房订阅这个服务
        4) 现在需要将这个服务迁移到B机房
        5) 此时需要先将这个服务在B机房扩容
        6) 然后再下掉A机房的服务
    问题就发生在这里，当下掉A机房服务后，消费者没有断开与A机房的连接，持续warn级别报警，
    等待15+分钟后，error级别异常报警，并显示连接拒绝

  • 问题分析

    通过源码分析，我们发现其根本原因在于RegistryDirectory.toInvokers方法中创建的invoker由CircuitBreakerInvoker封装了一层, 导致服务下掉后，仅仅清空了invoker缓存，但无法下掉invoker的连通检测线程

    • 检测线程是什么？

      当invoker创建后，会创建NettyClient去连接服务端，我们来看下NettyClient的父类AbstractClient中的构造方法
      AbstractClient的构造方法中会调用connect方法，connect方法中会调用initConnectStatusCheckCommand方法
      重点就在initConnectStatusCheckCommand方法中，这个方法内会启动一个周期调度线程，周期检测是否保持连接
      当销毁invoker时，需要调用AbstactClient.destroyConnectStatusCheckCommand方法中断这个检测线程
      但是由于CircuitBreakerInvoker中的destory方法并没有触发NettyClient的关闭，所以导致服务下掉后，检测线程还在一直运行，最终导致异常

    • Invoker创建链路

      RegistyDirectory
      notify -> refreshInvoker -> toInvokers - > new CircuitBreakerInvoker -> DubboProtocol.refer
      DubboProtocol
      refer -> new DubboInvoker -> new HeaderExchangeClient -> new HeaderExchangeChannel -> NettyTransporter(Netty3) -> new NettyClient -> new AbstractClient -> AbstractClient.connect
      NettyClient
      connect -> initConnectStatusCheckCommand

    • Invoker销毁链路

      RegistyDirectory
      notify -> refreshInvoker -> destroyAllInvokers -> CircuitBreakerInvoker.destory(继承父类AbstractInvoker.destory)
